# LocalPlayer V1.5 规格：批量分类与批量设置封面

**状态：已实施，等待 GitHub Actions 与真机验证**
**日期：2026-07-13**

## 目标

让用户在音乐库中一次选中多首已导入音乐，并在一次确认后为它们批量设置或清除分类、二级分类和自定义封面。此功能只修改 LocalPlayer 的 SwiftData 索引与 App 内封面数据，不修改 MP3 原文件、文件名或 MP3 标签。

## 已采用的假设

1. 批量选择只作用于当前筛选结果中被勾选的歌曲；切换分类筛选或退出编辑模式会结束当前选择。
2. 批量编辑页提供两个独立字段：分类和封面。每个字段均有“保持不变”“设置”“清除”三种状态；未选择的字段绝不被改动。
3. “设置分类”支持未分类、仅一级分类、一级 + 二级分类，以及输入新的一级/二级分类名称；规则与单曲编辑完全一致。
4. “设置封面”可从照片库或“文件”选择一张图片，该图片会复制为每首所选歌曲各自的 `artworkData`；“清除封面”将所选歌曲的封面设为 `nil`。
5. 一次确认可以同时应用分类与封面变更，提交前显示受影响歌曲数量；取消不修改任何歌曲。
6. 批量操作保存失败时恢复所有受影响歌曲的原分类与原封面，并显示错误；不允许出现只有部分歌曲成功更新的可见状态。
7. 当前正在播放的歌曲若被批量改封面，库、迷你播放器、正在播放页和锁屏信息在后续状态刷新后应使用新的封面；若只改分类，当前音频不中断。

## 用户流程

1. 用户在任意音乐库筛选页点“编辑”，歌曲行出现多选控件。
2. 用户选中一首或多首歌曲，界面显示“已选择 N 首”和“批量编辑”入口；未选歌曲时入口不可用。
3. 点“批量编辑”打开批量编辑页。用户可：
   - 保持分类不变、设置分类或设为未分类；
   - 保持封面不变、从照片/文件设置同一张封面或移除封面。
4. 用户点“应用到 N 首歌曲”，系统先显示确认对话框，明确将修改哪些字段和影响多少首音乐。
5. 确认后，App 验证图片数据、规范化分类、保存所有更改，并退出批量编辑状态。
6. 取消、图片无效、读取失败或保存失败时，歌曲原有分类和封面保持不变，并显示明确错误。

## 数据与接口契约

- 不新增 SwiftData 模型字段，不触发数据迁移。
- `Song.move(toCategory:subcategory:existingCategoryPaths:)` 继续作为分类规范化和二级分类约束的唯一模型入口。
- `Song.replaceArtwork(with:)` 和 `Song.removeArtwork()` 继续作为封面写入入口；批量封面数据必须先通过 `UIImage(data:)` 验证。
- 新增仅供批量编辑使用的值类型，例如：

```swift
enum BatchCategoryChange {
    case unchanged
    case move(categoryName: String?, subcategoryName: String?)
}

enum BatchArtworkChange {
    case unchanged
    case replace(Data)
    case remove
}
```

- 批量提交函数接收所选 `Song`、两类变更和已有分类路径；它先保存所有原始 `categoryName`、`subcategoryName`、`artworkData`，再统一修改并调用一次 `modelContext.save()`。
- 批量提交失败时，恢复上述快照；封面选择失败或取消时不进入提交函数。

## 技术栈与项目结构

- 平台：iOS 17.0、Swift 5、SwiftUI、SwiftData、PhotosUI、UniformTypeIdentifiers、UIKit。
- `App/LibraryView.swift`：编辑模式、多选状态、批量编辑入口、提交确认与错误提示。
- `App/BatchSongEditorView.swift`：分类选择、封面选择/预览、变更摘要和应用按钮。
- `Models/Song.swift`：复用现有分类、封面验证与写入规则；只在确有必要时增加小型批量辅助方法。
- `LocalPlayerTests/SongTests.swift`：批量分类、批量封面、未变更字段和失败恢复的单元测试。
- 不增加第三方依赖、网络服务、权限、MP3 写入或 SwiftData schema。

## 安全与数据完整性

- 图片和分类名称均视为外部输入：图片必须通过系统选择器的类型限制与 `UIImage(data:)` 双重验证；分类名称沿用现有去空格和规范化规则。
- 从“文件”选择图片时，只在安全作用域访问有效期间读取数据，并在完成后结束访问。
- 不记录、上传或提交用户音乐、图片、文件路径、IPA、签名材料或任何密钥。
- 任何批量清除、覆盖封面或移动分类都必须经过包含歌曲数量的显式确认，防止误操作。

## 代码风格

使用现有 Swift 命名、四空格缩进、`LocalizedError` 错误提示和 `#expect` 测试风格。批量编辑的变更值使用明确枚举，不使用含义模糊的可选值：

```swift
switch artworkChange {
case .unchanged:
    break
case .replace(let data):
    song.replaceArtwork(with: data)
case .remove:
    song.removeArtwork()
}
```

## 测试与验证

先写失败测试，再实现最小逻辑。测试至少覆盖：

- 对多首歌曲设置同一个一级/二级分类，且分类名复用现有规范化结果。
- 对多首歌曲设为未分类时，一级和二级分类都被清除。
- 批量设置有效图片后，每首选中歌曲保留该封面；未选歌曲不变。
- 批量移除封面只影响选中歌曲；“保持不变”不会修改原有封面或分类。
- 任一保存失败时，全部受影响歌曲恢复原始分类与封面。
- 空选择不能打开可执行的批量提交。

Windows 不具备 Xcode；提交后由 GitHub Actions 在 macOS 验证：

xcodegen generate
xcodebuild -project LocalPlayer.xcodeproj -scheme LocalPlayer \

## 边界

- **始终做：** 仅编辑显式选中的歌曲；显示影响数量并二次确认；验证图片；保存失败时恢复；保持当前音频播放不断。
- **先询问：** 批量重命名、批量删除、每首歌曲不同封面、封面裁剪/压缩、修改 MP3 内嵌标签、超过当前筛选范围的全库选择。
- **绝不做：** 修改、移动或删除来源 MP3；提交音乐、图片、IPA、证书、Apple ID 或密钥；在未确认时清除现有分类或封面。

## 成功标准

1. 用户能在当前筛选列表中多选歌曲，并清楚看到选择数量。
2. 用户能将同一分类路径一次应用于全部选中歌曲，也能一次设为未分类。
3. 用户能从照片库或文件选择一张有效图片并一次应用到全部选中歌曲，也能批量移除封面。
4. 未选歌曲和“保持不变”的字段不会被修改；取消或失败不会留下部分更新。
5. 批量操作不会中断当前音频播放，不会改变 MP3 原文件，也不会破坏二级分类规则。
6. GitHub Actions iOS 测试通过，且 iPhone 16 Pro 真机验证通过。

## 开放问题

无。批量重命名和批量删除容易造成误操作，明确不在本版本范围内；如需加入，应作为单独规格评审。
