# LocalPlayer V1.5 实施计划：批量分类与批量设置封面

**状态：已实施，等待 GitHub Actions 与真机验证**
**对应规格：**[V1.5 批量分类与批量设置封面规格](v1.5-batch-library-editing-spec.md)

## 概览

本版本为当前筛选音乐列表增加编辑模式、多选和批量编辑页。批量分类与批量封面通过一个显式变更请求共同提交：先验证、再保存一次 SwiftData；如保存失败则恢复全部受影响歌曲的原状态。整个过程不触碰 MP3 原文件，也不会中断正在播放的音频。

## 架构决定

1. 多选状态只保存在 `LibraryView`：使用歌曲 `UUID` 的集合，并在退出编辑模式或筛选切换时清空，确保操作范围始终可见。
2. 分类与封面变更用“未修改 / 设置 / 清除”的枚举表达，避免用 `nil` 同时表示“保持不变”和“移除”。
3. 先在内存中应用所有变更、保存一次 `ModelContext`；保存失败则用变更前快照恢复所有目标歌曲。
4. 封面文件读取复用已有 `ArtworkEditorSection` 的照片、文件、安全作用域和 `UIImage(data:)` 验证流程，不引入新依赖或新的图片访问路径。
5. 当批量操作包含当前歌曲的封面时，成功保存后显式刷新 `MPNowPlayingInfoCenter`，保证锁屏和控制中心更新而不重启或打断音频。

## 风险与缓解

| 风险 | 影响 | 缓解方式 |
| --- | --- | --- |
| 大量歌曲批量覆盖封面后误操作 | 高 | 仅显式选中歌曲可操作，确认对话框显示数量与字段摘要，未修改字段默认保持不变。 |
| SwiftData 保存失败留下部分内存状态 | 高 | 保存前记录全量快照；失败时立即恢复所有目标歌曲。 |
| 批量封面读取与单曲编辑规则不一致 | 中 | 复用或提取现有图片读取、类型限制、`UIImage(data:)` 验证与安全作用域逻辑。 |
| 当前正在播放歌曲的锁屏封面未更新 | 中 | 成功保存后仅在命中当前歌曲且封面变更时刷新 now-playing 信息。 |
| 编辑模式与播放按钮手势冲突 | 中 | 使用标准 `List(selection:)` 与 `EditButton`；编辑模式下优先完成选择，不触发歌曲播放。 |

## 实施边界

- 不修改 `project.yml`、CI、SwiftData schema 或 MP3 文件。
- 不加入批量重命名、批量删除、不同歌曲设置不同封面、图片裁剪或压缩。
- 每个任务均先写相应失败测试，再实现、审查、提交并由 GitHub Actions 验证。
