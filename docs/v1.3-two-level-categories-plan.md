# LocalPlayer V1.3 实施计划：两级音乐分类

**状态：已实施，等待 GitHub Actions 与真机升级验证**
**对应规格：** [V1.3 两级音乐分类规格](v1.3-two-level-categories-spec.md)

## 概览

把平铺分类升级为“一级 + 可选二级”。由于 `Song` 已持久化在用户的 SwiftData 存储中，先建立显式的 V1 → V2 轻量迁移，之后才接入分类规则和界面。每个阶段结束后都保持可构建状态。

## 架构决策

1. 保留 `categoryName` 为一级分类，增加可选 `subcategoryName`；旧记录自然变为仅一级分类。
2. 使用 `VersionedSchema`、`SchemaMigrationPlan` 和 V1 → V2 的 `MigrationStage.lightweight`，将 App 的 `ModelContainer` 改为使用该计划。
3. 分类仍由歌曲记录汇总，不引入 `Category` 模型，不保留空分类。
4. 一级筛选包含其全部子类；二级筛选只匹配完整的“一级/二级”路径，播放队列继续取当前筛选结果。

Apple 的 SwiftData 文档将版本化 schema 与迁移计划作为保存旧用户数据的路径；添加属性属于轻量迁移可处理的变更：

- https://developer.apple.com/videos/play/wwdc2023/10195/
- https://developer.apple.com/documentation/swiftdata/schemamigrationplan

## 任务清单

### 任务 14：建立 V1 → V2 SwiftData 迁移

**内容：** 将当前 `Song` 模型捕获为 V1 schema，建立含 `subcategoryName` 的 V2 schema 与轻量迁移计划；让 App 使用带迁移计划的 `ModelContainer`。

**验收：**

- [ ] 旧版记录可在 V2 打开，原一级分类、歌名、封面和 MP3 引用仍保留。
- [ ] 所有迁移后的旧记录 `subcategoryName == nil`。
- [ ] 新建歌曲可保存二级分类字段。

**验证：**

- [ ] 用临时持久化存储创建 V1 数据，再以 V2 容器打开并断言字段。
- [ ] GitHub Actions 完整 iOS 测试与归档通过。

**依赖：** 无。

**可能文件：**

- `Models/LocalPlayerSchema.swift`
- `Models/Song.swift`
- `App/LocalPlayerApp.swift`
- `LocalPlayerTests/PersistenceMigrationTests.swift`

### 任务 15：实现两级分类模型规则

**内容：** 为 `Song` 增加二级分类的规范化、路径比较、一级/二级汇总、移动、重命名和删除操作。

**验收：**

- [ ] 二级分类不能脱离一级分类存在；同一一级内的同名二级不会重复。
- [ ] 删除二级会使对应歌曲回到一级；删除一级会使其全部歌曲未分类。
- [ ] 一级或二级改名只更新对应范围的歌曲。

**验证：**

- [ ] 先添加 `SongTests` 的失败测试，再完成实现。
- [ ] GitHub Actions iOS 测试通过。

**依赖：** 任务 14。

**可能文件：**

- `Models/Song.swift`
- `LocalPlayerTests/SongTests.swift`

### 任务 16：实现歌曲编辑与移动的两级选择

**内容：** 将编辑歌曲页改为一级分类与依赖该一级的二级分类选择/输入；歌曲行的“移动到分类”菜单也显示两级结构。

**验收：**

- [ ] 用户可保存未分类、仅一级或一级/二级分类。
- [ ] 切换一级分类时，不会保留属于旧一级的二级分类。
- [ ] 保存失败会恢复歌曲原来的一级、二级、标题和封面数据。

**验证：**

- [ ] GitHub Actions iOS 测试通过。
- [ ] 真机编辑同一首歌的三种分类状态并重启 App。

**依赖：** 任务 15。

**可能文件：**

- `App/SongEditorView.swift`
- `App/LibraryView.swift`

### 任务 17：实现层级筛选与分类管理

**内容：** 将顶部分类菜单和“管理分类”改为一级/二级层级；父级筛选与子级筛选分别维护正确的播放队列。

**验收：**

- [ ] 管理菜单能重命名/删除一级及其指定二级，且遵循 V1.3 删除规则。

**验证：**

- [ ] GitHub Actions iOS 测试通过。

## 风险与缓解

| 风险 | 影响 | 缓解方式 |
| --- | --- | --- |
| 已有 SwiftData 存储无法直接升级 | 高 | 显式 V1/V2 schema 与轻量迁移；用临时 V1 存储测试，并在旧 IPA 上真机覆盖验证。 |
| 一级改变后保留了无效二级 | 中 | 在模型层强制二级依赖一级，界面仅展示当前一级的子类。 |
| 父级筛选与播放队列不一致 | 中 | 统一由当前筛选结果生成播放队列，并在筛选变化时同步。 |
| 分类操作逻辑过多挤在视图中 | 中 | 把名称规范化、匹配、批量改名和删除规则保留在 `Song` 模型层。 |
