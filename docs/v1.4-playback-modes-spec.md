# LocalPlayer V1.4 规格：顺序、随机与循环播放

**状态：已实施，等待 GitHub Actions 与真机验证**
**日期：2026-07-13**

## 目标

为本地音乐播放器提供明确的播放模式选择：顺序播放、随机播放、列表循环和单曲循环。用户可以在正在播放页切换模式；切换后，自动连播、界面“下一首”、锁屏和控制中心的“下一首”使用同一套规则。

## 已采用的假设

1. 默认模式为“顺序播放”，并在 App 重启后保留用户上次选择。
2. 顺序播放沿用当前行为：按当前筛选结果形成的播放队列从前到后播放，最后一首结束后停止。
3. 列表循环按当前队列顺序播放；最后一首自然结束或执行“下一首”时回到队列第一首。
4. 单曲循环在当前歌曲自然播放结束时，从 0 秒重新播放当前歌曲；手动“下一首”和锁屏/控制中心的下一首命令仍按当前队列顺序前进，不强制重复当前歌曲。
5. 随机播放只从当前播放队列中选择下一首，且不会紧接着重复当前歌曲；队列只有一首歌时，播放结束后停止。
6. 随机播放不在本版本实现“所有歌曲播完一次前绝不重复”的随机历史；“上一首”继续按照当前队列的原始顺序返回上一首。
7. 播放模式是全局偏好，不为每个分类、二级分类或单首歌曲分别保存。

## 用户流程

1. 用户在“正在播放”页看到当前播放模式按钮，默认显示顺序播放图标。
2. 点按按钮后，弹出菜单选择“顺序播放”“随机播放”“列表循环”或“单曲循环”；图标与辅助功能文字立即更新。
3. 顺序模式下，歌曲自然结束或用户点下一首时，播放器按当前队列顺序前进，到末尾停止。
4. 列表循环模式下，歌曲自然结束或用户点下一首时，播放器按当前队列顺序前进，到末尾后从第一首继续。
5. 单曲循环模式下，当前歌曲自然结束后立即重新播放；用户手动点下一首时仍可切换到队列中的下一首。
6. 随机模式下，歌曲自然结束、用户点下一首、或通过锁屏/控制中心点下一首时，播放器从当前队列中随机选择一首不同于当前歌曲的歌。
7. 用户切换分类或二级分类后，当前播放队列更新；后续切歌只在新的筛选结果中进行。

## 数据与接口契约

新增播放模式枚举：

```swift
enum PlaybackMode: String {
    case sequential
    case shuffle
    case repeatAll
    case repeatOne
}
```

- `AudioPlayerService` 是播放模式、当前播放队列和下一首决策的唯一来源。
- 播放模式使用 `UserDefaults` 保存，不修改 `Song`、不增加 SwiftData 模型字段，也不触发数据迁移。
- 随机选歌函数接收当前队列、当前歌曲 ID 和可注入的随机索引，返回下一首或 `nil`；它不得选择当前歌曲。
- 顺序、列表循环、单曲循环和随机模式均通过同一条“下一首决策”服务逻辑；自动连播、界面按钮和远程控制只传入不同的触发原因，避免多处行为不一致。
- 单曲循环只对“自然播放结束”触发；`nextTrackCommand` 和界面“下一首”不将当前歌曲当作下一首。

## 技术栈与项目结构

- 平台：iOS 17.0、Swift 5、SwiftUI、AVFoundation、MediaPlayer。
- `Services/AudioPlayerService.swift`：播放模式状态、持久化、四种模式的下一首决策、自然结束处理与远程命令统一入口。
- `App/NowPlayingView.swift`：展示和切换播放模式。
- `LocalPlayerTests/SongTests.swift`：覆盖顺序、随机、循环、单曲队列、模式持久化决策和自动连播的纯逻辑测试。
- 不增加第三方依赖、网络服务、权限或 SwiftData schema。

## 代码风格

继续使用现有 Swift 命名、四空格缩进和 `#expect` 测试风格；播放决策保持可测试的纯函数，避免在 SwiftUI 视图内复制判断：

```swift
static func nextSong(
    in songs: [Song],
    relativeTo currentSongID: UUID?,
    mode: PlaybackMode
) -> Song? {
    // 四种播放模式的唯一决策入口。
}
```

## 测试与验证

先写失败测试，再实现最小逻辑。测试至少覆盖：

- 顺序模式从当前歌曲得到队列中的后一首，最后一首返回 `nil`。
- 列表循环模式从最后一首返回队列第一首。
- 单曲循环模式在自然结束时返回当前歌曲；手动下一首仍返回队列中的后一首。
- 随机模式从多首队列中只返回非当前歌曲。
- 随机模式在单首队列中返回 `nil`。
- 自动连播和远程“下一首”均通过同一模式决策入口。
- 模式值可保存并在新建播放服务时恢复。

Windows 不具备 Xcode；提交后由 GitHub Actions 在 macOS 验证：

```bash
xcodegen generate
xcodebuild -project LocalPlayer.xcodeproj -scheme LocalPlayer \
  -destination "platform=iOS Simulator,id=<available-simulator-id>" \
  CODE_SIGNING_ALLOWED=NO test
```

真机验收：在 iPhone 16 Pro 导入至少三首歌，在“全部音乐”和一个二级分类中分别验证顺序、随机、列表循环、单曲循环的自动连播、下一首、锁屏下一首，以及重启后的模式保留。

## 边界

- **始终做：** 复用同一播放队列；不让随机选择当前歌曲；仅在列表循环模式回到第一首；为新决策逻辑添加回归测试。
- **先询问：** 完整随机历史、每个分类独立模式、在迷你播放器增加模式切换入口、修改锁屏 UI。
- **绝不做：** 提交 MP3、IPA、签名证书、Apple ID、密钥；修改 MP3 原文件或歌曲索引结构。

## 成功标准

1. 用户可在正在播放页切换顺序、随机、列表循环和单曲循环，并在重启 App 后看到上次选择。
2. 顺序模式的自动连播、界面下一首与远程下一首均严格按当前筛选队列前进，末尾停止。
3. 列表循环模式的自动连播、界面下一首与远程下一首均在队列末尾回到第一首。
4. 单曲循环模式只在自然播放结束时重播当前歌曲；手动下一首和远程下一首仍可切换歌曲。
5. 随机模式的自动连播、界面下一首与远程下一首均从当前筛选队列随机选择非当前歌曲。
6. 单首队列在随机模式下不会假装切歌或重复开始自身。
7. GitHub Actions iOS 测试通过，且 iPhone 16 Pro 真机验证通过。

## 开放问题

无。上述“随机不保存完整历史、上一首保持顺序返回”是本版刻意的范围边界；若你希望随机播放在一轮内绝不重复，或上一首也按随机历史返回，应在实施前改为独立需求。
