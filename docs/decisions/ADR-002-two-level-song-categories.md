# ADR-002：使用歌曲上的两个分类字段表示两级分类

## 状态

已接受，2026-07-13。

## 背景

V1.1 以 `Song.categoryName` 实现扁平分类。现在需要一级分类下的二级分类，同时必须保留已经存在的分类数据，且应用仍是仅供个人使用的本地播放器。

## 决策

保留 `Song.categoryName` 作为一级分类，并新增可选 `Song.subcategoryName` 作为二级分类。歌曲上的这两个可选字符串组成固定两级路径。

旧记录只设置了 `categoryName`，因此在新版本中自然成为“只有一级分类”的记录，无需将原有字符串迁移或修改 MP3 文件。

## 考虑过的替代方案

### 独立 `Category` SwiftData 模型

- 优点：可保存空分类、显式关系和将来扩展更多元数据。
- 缺点：引入模型关系、删除规则与迁移复杂度；对于两级、无空分类、仅个人使用的需求并无必要。
- 结论：不采用。

### 一个包含分隔符的路径字符串

- 优点：不增加字段。
- 缺点：分隔符可能出现在分类名中；重命名、去重和筛选容易出错，数据契约不清晰。
- 结论：不采用。

### 无限层级分类树

- 优点：未来任意扩展。
- 缺点：与已确认的两级需求不匹配，管理 UI、查询与迁移都会明显复杂化。
- 结论：不采用。

## 后果

- 一级分类改名会更新其全部歌曲，包括带二级分类的歌曲。
- 二级分类的改名和删除只影响对应的一级分类下的歌曲。
- 本 ADR 取代 [ADR-001](ADR-001-flat-song-categories.md) 中的扁平分类决策。
